# 클라이언트 UI 프레임워크 선택 근거

## 배경

`apps/client`는 Windows에서 상시 백그라운드로 도는 Python 트래커 프로세스의 상태를 모니터링하는 UI를 담당한다. 이후 필요하다면 클라이언트 UI 전반(설정, 리포트 뷰 등)까지 범위가 넓어질 수 있다.

요구사항:

- Python으로 구동되는 백그라운드 프로세스와 연결해 프로세스 현황을 모니터링할 수 있어야 한다.
- 인터넷이 안 되어 서버로 데이터를 업로드하지 못하는 상황에서도, 클라이언트는 계속 동작하며 SQLite에 로컬로 데이터를 쌓을 수 있어야 한다.
- "그냥 켜두면 자동으로 쌓이는" 가벼운 도구라는 프로젝트 철학([README](../README.md))에 맞게, 상시 상주 앱으로서 리소스 사용이 과도하지 않아야 한다.
- 이전에 Electron으로 목업을 구성해본 이력이 있다 (HTML/CSS/JS 자산 존재).

## 검토한 옵션

| 옵션 | UI 구현 방식 | Python 프로세스와의 연결 | 리소스/설치 용량 | 비고 |
|---|---|---|---|---|
| **Electron** | HTML/CSS/JS (Chromium 번들) | HTTP(FastAPI 로컬 서버) 또는 named pipe | 무거움 (~150-300MB 상주 메모리) | 지난 목업 자산 그대로 재사용 가능 |
| **Tauri** | HTML/CSS/JS (OS 내장 WebView2) | HTTP/IPC 별도 계층 필요 | 가벼움 (수십 MB대) | Rust 툴체인 학습/빌드 환경 추가 필요 |
| **pywebview** | HTML/CSS/JS (OS 내장 WebView2, Python 래퍼) | 불필요 — 같은 Python 프로세스/venv 안에서 직접 호출 | 가벼움 (런타임 별도 번들 없음) | 이전 Electron 목업의 웹 자산 재사용 가능 |
| **순수 Python GUI** (PySide6 / DearPyGui / Flet) | 각 프레임워크 자체 위젯 | 불필요 — 완전히 한 프로세스 | 프레임워크별 상이 | 기존 웹 자산은 버려야 함 |
| **WPF/.NET (C#)** | XAML | 언어가 달라 HTTP/gRPC/named pipe 등 IPC 필수 | 가장 가벼움, 네이티브 룩앤필 최고 | 트래커(Python)와 스택이 분리되어 유지보수 부담 증가 |

## 선택: pywebview

## 선택 이유

1. **IPC 계층 제거**: UI와 트래커 백그라운드 로직이 같은 Python 프로세스(또는 같은 venv 내 프로세스) 안에서 동작하므로, 프로세스 현황 모니터링을 위해 별도의 HTTP 서버나 named pipe를 구성할 필요가 없다. 상태 조회는 함수 호출/공유 객체로 처리 가능하다.
2. **SQLite 공유가 자연스러움**: 오프라인 캐싱 요구사항상 트래커가 이미 SQLite에 기록하고 있다면, 같은 프로세스의 UI 레이어가 동일한 커넥션/파일을 그대로 읽어 리포트를 그릴 수 있다. Electron/Tauri처럼 언어가 분리된 구조에서는 이 부분에 별도의 동기화/쿼리 계층이 필요해진다.
3. **기존 자산 재사용**: pywebview는 HTML/CSS/JS를 그대로 렌더링하므로, 지난 Electron 목업에서 만든 화면 코드를 상당 부분 재사용할 수 있다.
4. **상시 상주에 맞는 가벼움**: Chromium을 번들하지 않고 OS 내장 WebView2를 사용하므로, "그냥 켜두면 쌓이는" 백그라운드 상주 도구에 맞게 설치 용량과 메모리 사용이 Electron 대비 가볍다.

## 감수하는 트레이드오프

- WebView2 런타임이 대상 Windows 환경에 없으면 별도 설치가 필요할 수 있다 (다만 최신 Windows 10/11에는 기본 포함).
- Tauri/WPF 대비 UI 반응성이나 네이티브 룩앤필은 다소 떨어질 수 있다.
- 향후 클라이언트 UI가 크게 복잡해지고 별도 배포 채널(자동 업데이트, 코드사이닝 등)이 중요해지면 Tauri 등으로 재검토할 여지를 남겨둔다.

## 재검토 조건

- UI 요구사항이 단순 모니터링을 넘어 무거운 프론트엔드 앱 수준으로 커질 경우
- WebView2 의존성이 배포 대상 환경에서 실질적인 문제가 될 경우
- 트래커 프로세스와 UI 프로세스를 완전히 분리해야 할 이유(예: 권한 분리, 크래시 격리)가 생길 경우
